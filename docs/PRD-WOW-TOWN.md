## 1. 产品定位

全新久久小镇不是“多 Agent 管理后台”，而是一个让镇长一眼看到 AI 居民正在真实生活、交付、被评价并成长的个人 AI 小世界。

一句话：

> 镇长打开小镇的第一秒，就知道今天谁在忙、谁交付了、哪里卡住了、哪些作品值得看、哪位居民正在因为评价而进化。

## 2. 当前问题

基于现有数据库和界面观察：

- 当前任务状态：已完成 43，待处理 20。
- 当前评分状态：未评分 26。
- 现有界面偏“游戏地图 + 弹窗后台”，第一屏无法直接暴露治理重点。
- 小镇已经有任务、奏折、作品、积分、反馈沉淀，但用户必须点进市政厅多个 tab 才能理解闭环。
- 当前 Wow Moment 不应来自更花哨的地图，而应来自“这个小镇真的在运转”。

## 3. 北极星体验

### 3.1 第一屏必须回答四个问题

1. 今天小镇发生了什么？
2. 我现在最该处理什么？
3. 哪些居民正在成长？
4. 最近有什么值得打开的作品？

### 3.2 体验口号

> 不再是进入一个工具台，而是回到一个有人等你批奏折、有人等你评价、有人刚交付作品的小镇。

## 5. P0 需求

### P0.1 镇长驾驶舱

位置：第一屏顶部，替代单纯数字 topbar。  
目标：把“治理优先级”直接露出。

指标卡：

| 指标 | 计算方式 | 点击行为 |
|---|---|---|
| 待处理 | tasks.status 为 pending/processing | 打开市政厅悬赏板 |
| 未评分 | quests.status=done 且 rating 为空 | 打开交付 / 任务记录 |
| 今日交付 | completed 为今天的 done 任务 | 打开交付 / 任务记录 |
| 健康度 | 基于待处理和未评分粗略计算 | 打开奏折 |

健康度第一版规则：

```text
score = 100
if pending > 10: score -= 20
if unrated > 10: score -= 30
if pending > 20: score -= 20
if unrated > 20: score -= 20
最低 0，最高 100
```

### P0.2 第一屏行动建议

在地图右侧或顶部显示一句守护灵建议：

- 未评分 > 0：`有 N 件作品等你评价，居民会根据评价进化。`
- 待处理 > 0：`有 N 个悬赏正在排队，建议先看悬赏板。`
- 全部清空：`小镇运转顺畅，可以去作品长廊看看新成果。`

### P0.3 居民卡升级

居民列表从“头像 + 分数”升级为“状态 + 成长”。

每个居民展示：

- 姓名与角色。
- 当前状态：处理中 / 待命 / 需要评价反馈。
- 积分与成长等级。
- 最近任务标题（若有）。

### P0.4 市政厅入口分流

驾驶舱卡片点击后，直接打开市政厅对应 tab：

- 待处理 → 悬赏板。
- 未评分/今日交付 → 交付 / 任务记录。
- 健康度/奏折 → 奏折。

### P0.5 市政厅信息架构边界

市政厅不是多个“任务列表”的并列堆叠，而是小镇治理闭环的不同阶段。P0 必须避免“悬赏板、任务记录、探索”重复展示同一批内容。

#### 入口定义

| 入口 | 一句话定义 | 核心问题 | 主要内容 | 不应承载 |
|---|---|---|---|---|
| 待办 | 镇长当前最该处理事项的聚合入口 | 我现在该先处理什么？ | 待评分交付、待批奏折、积压悬赏、异常/超时、成长奖励机会 | 作为完整历史列表；替代其他 tab 的全部功能 |
| 悬赏板 | 镇长明确发起或采纳后进入执行的任务池 | 我想让居民做什么？这些任务执行到哪了？ | 发布悬赏、待分配任务、进行中任务、积压 pending/processing 任务 | 已完成历史账本；居民自发灵感；纯系统日志 |
| 交付 / 任务记录 | 已进入任务生命周期的事实账本 | 过去做了什么？交付了吗？评价了吗？ | done/failed/backlog 任务、产出链接、评分入口、评价记录、任务时间线 | 新任务发布作为主功能；未采纳探索灵感 |
| 奏折 | 北北整理后的正式治理建议 | 小镇哪里需要镇长决策？ | 北北早中晚奏折、健康报告、批准/驳回、批准后转悬赏 | 普通任务历史；居民随手实验 |
| 探索 | 居民或系统自发发现的机会池 | 小镇自己发现了什么值得试？ | 居民自发实验、灵感、可转悬赏机会、未成型观察 | 镇长已明确发布的任务；已完成任务账本 |
| 成长 / 智库 | 能力与资产沉淀入口 | 居民如何变强？哪些知识可复用？ | 积分、skill、反馈沉淀、资产、方法、模板 | 任务执行队列 |

#### 阶段关系

```text
探索：居民/北北自发发现机会
  -> 镇长采纳
奏折：北北正式治理建议
  -> 镇长批准
悬赏板：明确进入执行的任务
  -> 居民交付
交付 / 任务记录：形成事实账本，等待评分或复盘
  -> 评分沉淀
成长 / 智库：积分、skill、资产、方法沉淀
```

#### 状态归属规则

| 数据/状态 | 首要展示位置 | 可交叉引用位置 | 规则 |
|---|---|---|---|
| `pending` / `processing` 任务 | 悬赏板 | 待办、居民卡 | 表示已经进入执行队列，不属于探索 |
| `done_unrated` / `done` 且未评分 | 交付 / 任务记录 | 待办、镇长驾驶舱 | 首要动作是评分，不再回到悬赏板 |
| `done_rated` | 交付 / 任务记录 | 成长 / 智库、作品长廊 | 作为历史和资产沉淀 |
| `failed` / 超时 | 交付 / 任务记录 | 待办、奏折 | 需要复盘或重新发布时再转悬赏 |
| 北北建议 `suggestions.pending` | 奏折 | 待办 | 批准后创建任务并进入悬赏板 |
| 居民自发灵感 / 实验 | 探索 | 待办（仅高优先级） | 采纳后转成悬赏；未采纳不进入任务记录 |
| skill 奖励 / 反馈沉淀 | 成长 / 智库 | 待办、居民卡 | 不放入悬赏板，除非需要人工执行任务 |

#### 命名原则

- 前台文案使用“交付 / 任务记录”，避免只叫“任务记录”导致和悬赏板混淆。
- “探索”强调小镇主动性，不承诺一定执行。
- “奏折”强调北北正式治理建议，比探索更结构化。
- “待办”是聚合视图，不是新数据类型。

### P0.6 作品长廊入口

首页必须有显性入口：`作品长廊`。点击打开 `/gallery.html`。

## 6. P1 需求

- 居民详情抽屉：展示技能、代表作、最近评价。
- 小镇健康报告：由芝士或北北定期生成。
- 任务状态时间轴：发布→分配→处理中→交付→评分→沉淀。
- Skill 成长可视化：评价如何写入居民经验卡。

## 7. UI 设计原则

遵循久久审美：

- Typography-first，不靠色块堆叠。
- 默认色：米白 `#faf7f2`，炭黑 `#1a1a1a`，陶土红 `#c4553b`。
- 强调色面积小于 5%。
- 不用 emoji、渐变、玻璃态、假插图。
- 地图可以保留像素游戏感，但外围信息架构要更像高级产品驾驶舱。

## 8. 工程约束

- 优先使用现有 `/api/state` 数据，不新增复杂后端。
- 改 `town.html` 和 `town_server.py` 必须跑 `tests/test_all_features.py`。
- 若新增前端行为，先补测试锁定关键字符串/函数。
- 当前工作区有既有未提交改动，禁止重置或覆盖。
